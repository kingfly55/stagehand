# Milestone 7 — Agent Review Loop

## Scope
Build an automated PR review system: a Claude Code skill that reviews PRs for divergence impact, golden principle violations, test status, and description quality. Implement auto-labeling of PRs by divergence impact category. Integrate with the adversarial review pipeline from `docs/plans/completed/AGENT_ORCHESTRATION.md` (moved from root in Milestone 5).

**Prerequisites:** The `gh` CLI must be installed and authenticated for PR labeling and commenting. Document this requirement clearly in the scripts and skill file.

## Files to Create/Modify
- `.claude/skills/pr-review.md` — new; Claude Code skill for automated PR review
- `scripts/pr-review.sh` — new; script that performs automated PR checks
- `scripts/pr-label.sh` — new; script that determines and applies PR labels
- `.github/workflows/pr-review.yml` — new; GitHub Actions workflow for automated review on PR
- `claude.md` — add pointer to PR review skill

## Implementation Steps
1. **Create `scripts/pr-review.sh`:**
   - Must start with `#!/usr/bin/env bash`.
   - **Do NOT use `set -e`** at the top level — this script must always exit 0 (review summaries are informational), but individual commands like `pnpm typecheck` may fail. Instead, use `set -uo pipefail` and wrap each check in a function that captures the exit code:
     ```bash
     #!/usr/bin/env bash
     set -uo pipefail
     review_status=0  # track overall findings

     run_check() {
       local name="$1"; shift
       if "$@" 2>&1; then
         echo "✅ $name: PASS"
       else
         echo "❌ $name: FAIL"
         review_status=1
       fi
     }
     ```
   - Accept a PR branch name or ref as argument. Validate that the argument is provided; print usage and exit 1 if missing. **Note:** With `set -u`, accessing `$1` when no arguments are provided causes an immediate unbound-variable error before validation code runs. Use `${1:-}` for the check: `branch="${1:-}"; if [ -z "$branch" ]; then echo "Usage: ..."; exit 1; fi`.
   - Check `gh` CLI is available and set a flag for later checks: `has_gh=true; command -v gh &>/dev/null || { echo "WARN: gh CLI not found; skipping PR description check"; has_gh=false; }`. Use `$has_gh` to guard Check 5.
   - **Check 1 — Divergence impact**: `git diff --name-only native-base..<branch>` (use `native-base` not `main` — the fork's working branch is `native-base`) and cross-reference with `docs/fork/DIVERGENCE.md`. Report any upstream files being modified that aren't in DIVERGENCE.md. **Note:** If the branch ref doesn't exist locally, the script must `git fetch origin <branch>` first, or use `origin/<branch>`. **CI note:** In GitHub Actions, `actions/checkout` with `fetch-depth: 0` fetches all branches but only the PR ref is checked out. The script must `git fetch origin native-base` explicitly to ensure the base ref exists locally for the diff comparison.
   - **Check 2 — Golden principle violations**: Check if any changed files match protected paths from `docs/dev/GOLDEN_PRINCIPLES.md`. **Do NOT call `scripts/check-protected-paths.sh` directly** — it reads from `git diff --cached` (staged files) not branch diffs. Instead, replicate the protected-path matching logic against the output of `git diff --name-only native-base..<branch>`, applying the same exceptions (`page.ts`, `native/**`).
   - **Check 3 — Typecheck**: Run `pnpm typecheck` and report pass/fail. Capture output for the review summary.
   - **Check 4 — Tests**: Run `pnpm build:esm && pnpm test:core` and report pass/fail. **Note:** This is expensive (~30s+). Consider making it optional with a `--skip-tests` flag, since CI will run these checks independently.
   - **Check 5 — PR description quality**: Use `gh pr view <branch> --json body -q .body` (note: `gh pr view` accepts a branch name). Handle case where `gh` is not authenticated or no PR exists for the branch gracefully (skip this check with a warning).
   - Output a structured review summary in markdown format.
   - **Exit code:** Always `exit 0` explicitly at the end of the script (review summaries are informational). Use the summary content and `$review_status` to indicate pass/fail in the output, not the script's exit code — otherwise CI workflows will fail the job on review findings. The `set -o pipefail` may cause non-zero exits from intermediate pipe commands, so ensure the final line is `exit 0`.
   - Make executable.

2. **Create `scripts/pr-label.sh`:**
   - Must start with `#!/usr/bin/env bash` and `set -euo pipefail`.
   - Accept a PR number as argument. Validate that the argument is provided and is a positive integer. **Note:** With `set -u`, use `${1:-}` to safely access the argument: `pr_num="${1:-}"; if ! [[ "$pr_num" =~ ^[0-9]+$ ]]; then echo "Usage: ..."; exit 1; fi`.
   - Check `gh` CLI is available and authenticated: `gh auth status &>/dev/null || { echo "ERROR: gh not authenticated"; exit 1; }`
   - Determine the label based on changed files:
     - `upstream-safe`: No files in `docs/fork/DIVERGENCE.md` are touched, no protected paths touched.
     - `divergence-update`: Modifies files listed in DIVERGENCE.md or adds new fork-specific files.
     - `needs-human-review`: Touches protected paths (`understudy/**`, `handlers/**`), modifies golden principles, or modifies CI/workflow files (`.github/workflows/**`).
   - **Create labels if they don't exist** before applying (use `gh label create --force` which is idempotent):
     ```bash
     gh label create "upstream-safe" --color "0E8A16" --description "No divergence impact" --force
     gh label create "divergence-update" --color "FBCA04" --description "Updates fork divergence" --force
     gh label create "needs-human-review" --color "D93F0B" --description "Touches protected paths" --force
     ```
   - Apply the label via `gh pr edit --add-label`.
   - Make executable.

3. **Create `.github/workflows/pr-review.yml`:**
   - Trigger on `pull_request` events (opened, synchronize) targeting `native-base` branch.
   - Required permissions (explicit in workflow):
     ```yaml
     permissions:
       contents: read
       pull-requests: write
       issues: write
     ```
   - Steps:
     1. Checkout code with `fetch-depth: 0` (needed for `git diff` against base branch)
     2. Fetch `native-base` explicitly (checkout only gives us the PR branch):
        ```yaml
        - name: Fetch base branch
          run: git fetch origin native-base
        ```
     3. Setup Node.js and pnpm (match versions from existing CI: check `.github/workflows/ci.yml` for reference)
     4. Install dependencies (`pnpm install --frozen-lockfile`)
     5. Run `scripts/pr-review.sh "${{ github.head_ref }}"` and capture output to a file:
        ```yaml
        - name: Run PR review
          run: bash scripts/pr-review.sh "${{ github.head_ref }}" > /tmp/review-output.md 2>&1 || true
        ```
     6. Run `scripts/pr-label.sh "${{ github.event.pull_request.number }}"` with the PR number (this can fail without breaking the workflow — wrap in `|| true` or use `continue-on-error: true`)
     7. Post the review summary as a PR comment via `gh pr comment "${{ github.event.pull_request.number }}" --body-file /tmp/review-output.md`
   - **Use `GITHUB_TOKEN`** (available by default) for `gh` CLI auth — set `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` as a job-level env var.
   - **Concurrency:** Add `concurrency: { group: pr-review-${{ github.event.pull_request.number }}, cancel-in-progress: true }` to avoid duplicate reviews on rapid pushes.
   - **Comment deduplication:** Consider using `gh pr comment --edit-last` to update existing review comments rather than posting new ones on every push (requires the workflow to have posted at least one comment previously; fall back to `--body-file` on first run).

4. **Create `.claude/skills/pr-review.md`:**
   - Skill name: `pr-review`
   - Description: Review a PR for divergence impact, golden principle compliance, and quality.
   - Instructions:
     1. Identify the PR branch (from context or argument).
     2. Run `bash scripts/pr-review.sh <branch>`.
     3. Review the output and add any additional observations about code quality, architecture alignment, or potential issues.
     4. If golden principle violations are found, flag them prominently and suggest remediation.
     5. Determine the appropriate label and explain why.
     6. Post findings as a PR comment.
   - References: `docs/dev/GOLDEN_PRINCIPLES.md`, `docs/fork/DIVERGENCE.md`, `docs/fork/MERGE_POLICY.md`.

5. **Update `claude.md`** to reference the PR review skill and workflow.

6. **GitHub labels** are created automatically by `scripts/pr-label.sh` using `gh label create --force` (idempotent). No manual setup needed. The labels are:
   - `upstream-safe` (green) — No divergence impact
   - `divergence-update` (yellow) — Updates fork divergence
   - `needs-human-review` (red) — Touches protected paths

## Verification Steps
1. Syntax-check all scripts:
   ```bash
   bash -n scripts/pr-review.sh && echo "OK: pr-review.sh" || echo "FAIL"
   bash -n scripts/pr-label.sh && echo "OK: pr-label.sh" || echo "FAIL"
   ```
2. Validate the workflow YAML:
   ```bash
   # PyYAML may not be installed; fall back to structural check
   python3 -c "import yaml; yaml.safe_load(open('.github/workflows/pr-review.yml'))" 2>/dev/null && echo "OK: YAML valid" \
     || { grep -qE '^"?on"?:' .github/workflows/pr-review.yml && grep -q "^jobs:" .github/workflows/pr-review.yml && echo "OK: structural" || echo "FAIL"; }
   ```
   **Note:** GitHub Actions YAML commonly uses `"on":` (quoted) because `on` is a YAML reserved word. The check uses `^"?on"?:` to match both `on:` and `"on":`.
3. Confirm the review script checks all required areas:
   ```bash
   count=$(grep -c "DIVERGENCE\|GOLDEN_PRINCIPLES\|typecheck\|test:core\|description" scripts/pr-review.sh)
   [ "$count" -ge 5 ] && echo "OK: $count check references found" || echo "FAIL: only $count (expected >= 5)"
   ```
   Also confirm the script does NOT use `set -e` (which would cause early exit on check failures):
   ```bash
   if grep -q '^set -e' scripts/pr-review.sh || grep -q '^set -euo' scripts/pr-review.sh; then
     echo "WARN: pr-review.sh uses set -e which will cause early exit on pnpm typecheck failure"
   else
     echo "OK: no set -e"
   fi
   ```
4. Confirm the label script supports all three labels:
   ```bash
   grep -c "upstream-safe\|divergence-update\|needs-human-review" scripts/pr-label.sh
   # Should be >= 3
   ```
5. Confirm skill file exists and references key docs:
   ```bash
   grep -c "GOLDEN_PRINCIPLES\|DIVERGENCE\|MERGE_POLICY" .claude/skills/pr-review.md
   # Should be >= 3
   ```
6. Run `pnpm typecheck` and confirm zero errors

## Agent Notes
- The `gh` CLI must be installed and authenticated. On Ubuntu: `sudo apt-get install gh` (if the GitHub CLI apt repo is configured) or `sudo snap install gh`. On macOS: `brew install gh`. Then authenticate: `gh auth login`.
- The workflow uses `GITHUB_TOKEN` which has limited permissions. If the fork is private, ensure the token has `pull-requests: write` permission.
- `scripts/pr-review.sh` should NOT exit non-zero on review findings — only on script errors. Review findings should be communicated via the output/comment, not exit codes, to avoid failing the CI job.
- The `git diff --name-only native-base..<branch>` comparison in `pr-review.sh` uses `native-base` as the base, not `main`. If PRs target `main`, update accordingly.
